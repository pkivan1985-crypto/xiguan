## 一、上级规则与冲突处理

本文件约束正式 DoHabit Web/PWA 代码仓库中的 Codex 主 Agent 和所有子 Agent。

权威优先级：

1. 用户在当前对话中的最新明确指示。
2. 上级工作区 `../AGENTS.md`。
3. `../00-项目统筹/01-产品需求与策划/01-APP需求文档3.0.md`。
4. `../00-项目统筹/01-产品需求与策划/02-详细策划案3.0.md` 与用户确认的 Figma/FigJam。
5. `../00-项目统筹/02-技术与架构/01-技术栈与发布冻结.md`、`../00-项目统筹/02-技术与架构/02-项目架构基线.md` 及详细架构设计。
6. 本 Web 工程宪法。
7. 当前任务说明、代码附近约定和自动化测试。
8. DoHabit 原功能和默认行为。

上层规则与本文件冲突时以上层为准。本文件更接近代码不代表可以覆盖产品决定。发现冲突必须写明“当前规则、冲突来源、建议处理”，不得静默混合。

## 二、固定项目上下文

- 产品：重复成果显化。
- 形态：公开可访问、手机优先响应式网页 + PWA；暂不开发原生 App。
- 用户：第一阶段单机单用户，无注册、无登录。
- 数据：只保存在用户浏览器；无后端、无云同步、无远程数据库。
- 许可证：项目整体 AGPL-3.0。
- 主底座：DoHabit 是唯一主代码底座，必须保留 Git 历史、LICENSE、作者声明和修改记录。
- 技术：React 19.2.7、TypeScript 6.0.3、Vite 8.1.4、React Router 7.18.1、Zustand 5.0.14、Dexie 4.4.4、vite-plugin-pwa 1.3.0。
- 动画：Framer Motion 12.42.2 + Lottie Web 5.13.0 + CSS 动画。
- 旧数据：旧 `app/` 与 DoHabit 原数据均为测试数据，不开发旧数据迁移器。
- 正式托管：GitHub 公共仓库 + Cloudflare Pages 静态托管；第一阶段不用 Functions、Workers、D1、KV、R2。

开始新任务、上下文压缩或中断恢复时：

5. 读取任务涉及的源文件、导出、直接调用方、相邻实现和相关测试。
6. 检查 `git status`、当前分支和未提交修改；用户已有修改不得覆盖。
7. 先汇报目标、范围、当前证据、风险和成功标准，再开始写入。

不得只依赖聊天记忆或 Hindsight 代替当前文件事实。Hindsight 项目 bank 为 `repeat-achievement-manifestation`，仅记录已验证的长期决定和任务结论，不保存秘密或聊天全文。

## 四、Agent 自主权限

### 必须停下并询问用户

- 新增、取消或显著改变产品功能。
- 改变公开无账号、本地数据、无后端或无云同步边界。
- 引入付费服务、账号系统、分析追踪、广告、在线 AI 或远程数据库。
- 改变 AGPL-3.0、DoHabit 主底座或正式发布渠道。
- 删除真实数据、旧 `app/`、Git 历史、许可证文件或来源记录。
- 未经确认改变页面主流程、信息层级或用户已经批准的画面。
- 推送远程仓库、合并 `main`、创建公开 Release 或部署生产环境。

### 模块批准后可自主决定

- 符合既定架构的文件拆分、函数命名和局部实现方式。
- 测试文件组织、mock/fake 选择和失败注入方式。
- 不改变产品含义的错误提示、空状态和无障碍文案微调。
- 为修复当前任务而做的最小局部重构。
- 同主版本内、已经单独验证且不改变锁定技术边界的补丁调整。

自主执行不等于扩大任务范围。任何写入都必须服务于当前模块成功标准。

## 五、不可违反的产品规则

1. 大类不是循环卡，不能进入今日卡槽。
2. 用户卡来自官方循环卡模板，并属于我的卡套。
3. 今日区域固定 6 个槽位，只能放用户卡。
4. 今日卡每天由用户手动逐张选择，不自动继承昨天。
5. 行动记录是进度、月历和目标状态的事实来源。
6. 同一用户卡同一本地自然日最多一条累计行动记录。
7. 今日成果必须整批事务保存成功后才播放。
8. 动画失败、跳过或减少动态效果不能撤销行动记录。
9. 成果月历某天有至少一条有效记录时只显示一个大勾。
10. 阶段目标支持 `quantity`、`activeDays`、`both`，跑步默认 `quantity`。
11. 第一阶段不颁发正式勋章，只允许保留完成快照接口。
12. 不存在长期快捷卡槽，不保留 `slotIndex` 或永久 SlotAssignment。
13. 运行时不调用 AI，只使用确定性规则和已登记预制素材。
14. 不兼容或迁移旧测试数据，不得为此增加生产复杂度。

## 六、目录与依赖规则

沿用 DoHabit Feature-Sliced 结构：

```text
src/app
src/pages
src/widgets
src/features
src/entities
src/shared
```

依赖方向为 `app/pages → widgets → features → entities → shared`。

- `shared` 不导入 entities、features、widgets 或 pages。
- `entities` 不导入 features、widgets 或 pages。
- `features` 不导入 widgets 或 pages。
- 页面只装配流程，不复制领域规则或直接操作数据库。
- React 组件不直接访问 Dexie。
- 业务计算使用纯 TypeScript，不依赖 React、DOM、动画或 Service Worker。
- 新增卡片类型优先增加模板、单位、校验和素材配置，不复制页面与保存流程。
- 不为单次使用建立通用框架，不做与当前模块无关的重构。

## 七、数据、状态与事务

### 数据归属

- Dexie/IndexedDB：用户卡、目标、目标修订、今日草稿、行动记录、成果批次和业务设置。
- Zustand：弹窗、筛选、表单步骤、临时选择、播放器位置等界面会话状态。
- localStorage：语言、减少动态效果和可丢失提示状态。
- Cache Storage：可重新获取的应用壳和素材，不保存业务事实。

Zustand 不得成为业务事实的唯一副本。刷新页面后，业务状态必须可从 Dexie 恢复。

### 事务规则

- `save-today-outcome` 是今日成果唯一跨表提交入口。
- 行动记录、目标修订/完成快照和不可变 OutcomeBatch 必须在同一 Dexie 读写事务中提交。
- 任一步失败全部回滚，并保留用户草稿。
- 使用稳定 `submissionId` 防止重复提交。
- `actionRecords` 使用 `&[userCardId+localDate]` 唯一复合索引或等价数据库约束。
- 禁止通过清空数据库、删除索引或拆分事务来绕过失败测试。
- schema 从 v1 开始；后续变化通过 Dexie version migration，不能要求用户清库升级。

### 导入导出

- 只支持 3.0 自身带 `schemaVersion` 的备份。
- 导入顺序固定为解密、解析、格式校验、关系校验、内存升级、预览确认、单事务替换、回读验证。
- 校验完成前不得清空现有数据。
- 导入失败不得污染现有数据库。

## 八、页面、动画、PWA 与无障碍

- 页面布局开发前必须有用户确认的草图或 Figma 低保真。
- 关键页面必须在 360、390、430px 视口无横向滚动、遮挡和不可点击区域。
- Framer Motion 只负责页面和组件编排，Lottie 只负责预制素材播放，CSS 负责轻量反馈。
- 动画组件不写业务数据；每个成果动画必须有静态图和文字兜底。
- 遵守 `prefers-reduced-motion`，关键信息不能只靠运动或颜色传达。
- 首次成功加载后，核心跑步闭环必须可断网使用。
- Service Worker 更新不能中断正在进行的 Dexie 事务；失败时继续运行旧版本。
- PWA 缓存升级不得删除 IndexedDB。
- 交互组件必须有可访问名称、键盘/焦点行为和足够触控区域。

## 九、实现与测试流程

详细阶段划分、开工复述模板、三组验收问题、证据格式和阶段检查点以 `../00-项目统筹/03-Agent治理/02-分阶段开发与证据验收规范.md` 为准。

M0—M2 是工程门禁，允许暂时没有完整用户流程，但必须有可重复工程证据；M3—M8 组合为 R1—R3 用户版本，每个版本必须形成真实可用闭环。只允许详细拆分当前已获批阶段。

每个代码任务按以下顺序执行：

```text
读取设计与相关代码
→ 写明成功标准和禁改范围
→ 先添加会失败的领域/用例测试
→ 运行并确认失败原因正确
→ 编写最小实现
→ 运行相关测试
→ 运行 lint、完整测试和 build
→ 需求符合性审查
→ 代码质量审查
→ 浏览器/PWA/手机视口验收
→ 文档和来源台账回写
```

- 共享业务逻辑、事务、bug 修复和用户可见行为变化必须有测试。
- 测试应证明业务意图，不只断言当前 DOM 形状。
- 不得删除、跳过或放宽有效测试来让构建变绿。
- 命令失败必须报告原始失败域，不把环境、网络和代码问题混为一谈。
- “测试通过”只指本轮实际运行并检查过的命令。
- M0 后以 `package.json` 和 lockfile 登记的实际命令为准；Windows PowerShell 优先使用 `npm.cmd`。

## 十、Git、许可证和外部来源

- 保留 DoHabit 原 Git 历史、LICENSE 和版权声明。
- 项目组合代码按 AGPL-3.0 发布，应用设置页保留源代码和许可证入口。
- 不直接提交到受保护的 `main`；实现使用边界清晰的任务分支和可审查提交。
- 不执行 `git reset --hard`、强制推送、历史重写或删除用户分支，除非用户明确授权。
- 外部代码、素材、字体、动画和音频进入构建前必须登记 `../00-项目统筹/04-开源与合规/01-开源代码来源与修改台账.md`。
- 未确认许可证、限制商用或来源不明的素材不得进入发布构建。
- 密钥、令牌、密码、cookie 和个人数据不得进入代码、日志、提交或 Hindsight。
- 未经用户明确授权，不推送远程、不合并 `main`、不创建 Release、不部署生产环境。

## 十一、多 Agent 与写入隔离

- 主 Agent 负责用户沟通、任务拆分、冲突裁决、协调状态和最终报告。
- 实现、需求审查、质量审查和 QA 尽量使用不同上下文。
- 修改相同文件、同一 Dexie schema 或同一事务链路的任务必须串行。
- 只有互不写同一状态的只读研究、测试分析或独立模块才可并行。
- 子任务必须给出目标、背景、允许写入、禁改范围、验收标准、验证命令和结果格式。
- 子 Agent 建议不会自动成为正式规则；主 Agent 必须对照权威文档审核。
- 用户已有未提交修改归用户所有，不覆盖、不清理、不顺手重构。

## 十二、完成定义

任务只有在以下条件全部满足时才可报告完成：

- 只实现已批准范围，没有推测性功能。
- 产品规则、架构依赖和数据边界一致。
- 相关测试、lint、build 已实际运行并通过；不能运行的项目已明确说明。
- 数据事务、异常和失败路径得到相称验证。
- 需求审查、代码质量审查和必要的浏览器验收没有未解决阻塞项。
- 文档、版本、来源台账和当前实现一致。
- 没有未经授权的推送、发布、删除或外部状态变更。

标准交付报告：

```text
已完成：用户现在能看到和使用什么
已验证：实际运行的测试、构建和浏览器路径
未完成/风险：跳过、失败、环境限制和剩余事项
文档/合规：更新的规则、来源和许可证记录
下一步：需要用户确认的下一个模块
```

## 十三、换号接力与单 Agent 降级

- 同机更换 Codex 登录账号时，必须遵守 `../00-项目统筹/06-换号接力/02-换号交接操作规范.md`。
- 新账号第一次只能读取总索引、根/Web 宪法、当前接力卡、协调状态和当前实施基线，并向用户汇报；用户确认接管前不得写入。
- 两个账号不得同时修改仓库。旧账号标记 `READY_FOR_TAKEOVER` 后保持只读，新账号获批后标记 `ACTIVE`。
- 新账号必须检测子 Agent、插件、连接器、模型和终端能力，不得假设与旧账号相同。
- 没有子 Agent 时必须启用 `SINGLE_AGENT_FALLBACK`，执行 `../00-项目统筹/06-换号接力/03-单Agent降级执行规范.md`。

- 接力卡和 Hindsight 不能替代实际文件、当前实施基线、Git 状态、进程和测试证据。
- 接力文件、日志、提交和 Hindsight 均不得保存秘密或账号身份信息。
